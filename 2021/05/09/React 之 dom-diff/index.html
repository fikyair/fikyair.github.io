React 之 dom-diff

2021-05-09

标准的 dom-diff 算法时间复杂度是 O(n^3)，显然性能在某些情况下是无法达到要求的，Facebook 工程师将其时间复杂度降到了 O(n)。本文通过不同的 case 来对其思想做进一步的整合。

DOM-DIFF算法

在 React17+ 中 DOM-DIFF 就是根据老的 fiber树 和最新的 JSX 对比生成新的 fiber树 的过程

一般来说，一个老的树变成一个新的树，算法实现的时间复杂度是 O(n^3)，性能较差

React优化原则

  • 只对同级节点进行对比，如果 DOM 节点跨层级移动，则 React 不会复用
  • 不同类型 type 的元素会产出不同的结构，会销毁老结构，创建新结构
  • 可以通过 key 标识移动的元素

单节点

  • 如果新的节点只有一个的话

case 1

新旧节点 type不同，如果节点没有写 key，key 就是索引 （key=“0”）

在 commit 阶段，remove h1，appendChild h2

case 2

新旧节点 type 和 key 都不一样

只需 h1 标记为删除
h3 h4 标记为删除

case 3

type和key都相同

如果对比后发现新老节点一样的，那么会复用老节点，复用老节点的DOM元素和Fiber对象

再看属性有无变更，如果有变化，则会把此Fiber节点标准为更新

h1.innerHTML = 'h1-new';

case 4

key 相同但是 type 不同，直接删除所有老节点

如果key相同，但是type不同，则不再进行后续对比了，

直接把老的节点全部删除

插入新节点即可

多节点

  • 如果新的节点有多个节点的话
  • 节点有可能更新、删除、新增
  • 多节点的时候会经历二轮遍历
  • 第一轮遍历主要是处理节点的更新,更新包括属性和类型的更新
  • 第二轮遍历主要处理节点的新增、删除和移动
  • 移动时的原则是尽量少量的移动，如果必须有一个要动，新地位高的不动，新地位低的动

如果新的节点有多个的话
我们经过二轮遍历

  1. 第一轮处理更新的情况 属性和类型 type 的更新，更新或者说保持，不变的频率会比较高，
    若处理 删除/移动 的话，节点的位置会有变化，不利于 diff

  2. 第二轮处理新增 删除 移动 的情况

case1

一一对比，都可复用，只需更新

最后会得到一个操作步骤：

  1. 更新A
  2. 更新B
  3. 更新C
  4. 更新D

case2

一一对比，key 相同，type 不同，删除老的，添新的

删除老 li A
插入 div A
更新 B 更新 C 更新 D

case3

更新A

key 不一样，说明可能有位置变化

如果第一轮遍历的时候，发现 key 不一样，则立刻跳出第一轮循环

下面详细说一下：

在 react 源码中，每个操作都有一个标识，称之为 Effect.Tag

更新为：

lasPlacedIndex 最后放置的老的节点的挂载的索引位置

oldIndex 节点在老集合中的索引位置

newIndex 节点在新集合中的索引位置

  1. 第一轮循环
    type 和 key 都一样 A 能复用，更新 A 就可以，
    然后比较第二个元素，Key 不一样，则马上跳出第一轮循环，进入第二轮循环

  2. 第二轮循环
    新建 map： key 就是元素的 key，值就是老的 fiber 节点 let map = {'B':B,'C':C,'D':D,'E':E,'F':F};
    每当处理完一个节点，就从 map 中删除，
    等新的 JSX 数组遍历完之后，map = {'D':D,'F':F}，把 map 里面的 Fiber 节点全部标记为 Deletion，也就是 D 和 F ;

继续遍历新的节点

  • 当遍历到 C 的时候，从 老的集合 中找，是否能找到 C

    发现找到了，C 的 (oldIndex = 2) > (lastPlacedIndex = 0)

    C 的位置不需要移动。effectTag 标记为 Update，另外从 map 中删除 C，因为 C 可以复用

    lastPlacedIndex = 2，2 就是 C 在老集合中的位置 oldIndex

  • 当遍历到 E 的时候，从 老的集合 中找，是否能找到 E

    发现找到了，E 在老集合中的位置 (oldIndex = 4) > (lastPlacedIndex = 2)

    E 的位置不需要移动，effectTag 标记为 Update

    lastPlacedIndex = 4，4 就是 E 在老集合中的位置 oldIndex

  • 当遍历到 B 的时候，从 老的集合 中找，是否能找到 B

    发现找到了，B 在老集合中的位置 (oldIndex = 1) < (lastPlacedIndex = 4)，

    辛苦挪动下位置（因为 B 在新的集合中的位置比较靠后）

    B 向后移动到 E 的后面，effectTag 标记为 PlacementAndUpdate

  • 当遍历到 G 的时候，从 老的集合 中找，是否能找到 G

    没有找到，effectTag 标记为 Placement

  • 发现 map 集合中还有 D 和 F，但新的集合中没有 D 和 F，effectTag 标记为 Deletion

在 commit 阶段，先执行 Deletion 标记的操作，再执行其他操作。为什么？

diff 阶段可以暂停，没有时间片了就交还控制权。

更新为：

可以看到，我们以为从 abcd 变为 dabc，只需要将 d 移动到前面。

但实际上 React 保持 d 不变，将 abc 分别移动到了 d 的后面。

从这点可以看出，考虑性能，我们要尽量减少将节点从后面移动到前面的操作。

为什么不要用索引作为key

如果用索引对比的话

A-B
B-C
C-D
D-A

删除所有老 DOM，全部添加新 DOM

如果用 key 来比较，只需移动 A 到最后就行了。
